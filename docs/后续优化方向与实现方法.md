# AI 别墅沙发试摆助手：后续优化方向与实现方法

## 1. 优化目标

当前方案以“房间图 + 沙发图 + 提示词”的端到端生图为主。它能快速出图，但无法稳定控制沙发的位置、朝向、尺度、原有家具处理和通道避让。

后续目标是把流程升级为“可控摆位 + 局部 AI 精修”：

- 用户的自然语言要求为最高优先级。
- 未指定要求时，模型自动选择合理摆位。
- 沙发本体尽量保持与商品图一致。
- 位置、朝向、大小和家具保留/移除可解释、可验证。
- AI 只负责它最擅长的真实感融合：阴影、反射、遮挡、边缘和环境光。

## 2. 推荐架构

```text
房间图 + 沙发图 + 用户要求
        │
        ├─ A. 场景理解
        │     └─ 识别地面、墙面、门窗、屏幕、旧沙发、通道、视觉焦点
        │
        ├─ B. 沙发提取
        │     └─ 生成透明沙发图（RGBA）和沙发遮罩
        │
        ├─ C. 意图解析与候选摆位
        │     └─ 用户要求结构化 + 多模态模型给出候选地面区域
        │
        ├─ D. 程序化摆位
        │     └─ 透视缩放、旋转、碰撞过滤、通道避让、初版合成图
        │
        ├─ E. 局部 AI 精修
        │     └─ 仅修复沙发周围区域、旧家具移除区域、阴影与反射
        │
        └─ F. 结果质检
              └─ 检查是否满足用户要求，不满足则带纠正指令重试
```

## 3. 具体实现方法

### 3.1 场景理解

使用多模态模型分析房间图，但输出必须是严格 JSON，而不是一段描述文字。

建议输出：

```json
{
  "floorPolygon": [[0.08, 0.61], [0.92, 0.61], [0.98, 0.98], [0.03, 0.98]],
  "vanishingDirection": "center",
  "anchors": [
    { "id": "projection_screen", "type": "screen", "box": [0.34, 0.23, 0.62, 0.58] },
    { "id": "stairs", "type": "obstacle", "box": [0.73, 0.31, 0.96, 0.88] }
  ],
  "walkways": [
    { "polygon": [[0.68, 0.58], [0.96, 0.58], [0.96, 0.98], [0.68, 0.98]] }
  ],
  "existingFurniture": [
    { "id": "existing_sofa", "type": "sofa", "box": [0.34, 0.55, 0.68, 0.82] }
  ]
}
```

坐标统一使用 0 到 1 的归一化坐标，便于响应式图片尺寸和后端处理。

### 3.2 沙发抠图

沙发图不能只作为参考图交给生图模型。应先输出：

- 透明背景沙发图（PNG/WebP RGBA）
- 沙发遮罩（Mask）
- 主体包围盒
- 推荐观察角度

实现选择：

- 使用 SAM 2 进行主体分割。
- 对边缘使用 Matting 模型做细化，避免靠垫、扶手、脚部和镂空区域被切坏。
- 若商品图背景复杂，进入“请确认沙发主体”的轻量交互，而不是直接生成。

### 3.3 用户意图解析

用户自然语言要求必须优先于默认逻辑。将文本解析成结构化约束：

```json
{
  "placement": {
    "relation": "opposite",
    "target": "projection_screen",
    "facing": "projection_screen"
  },
  "furniture": {
    "remove": ["existing_sofa"],
    "preserve": ["coffee_table", "rug"]
  },
  "constraints": ["keep_right_walkway_clear"],
  "sourceText": "移除原有黑色沙发，保留茶几和地毯，新沙发朝向投影幕布，右侧留出通道"
}
```

规则：

- 用户明确提到“保留”“移除”“替换”“朝向”“旁边”“对面”“避开”时，写入硬约束。
- 用户未指定位置时，再由场景模型生成候选位。
- 用户要求与房间结构冲突时，不擅自忽略，而是给出可执行的替代位置。

### 3.4 候选摆位与程序化摆放

候选位可以由多模态模型提出，但必须经过程序过滤。

每个候选位需要包含：

```json
{
  "id": "candidate_1",
  "floorPolygon": [[0.22, 0.62], [0.74, 0.62], [0.80, 0.89], [0.16, 0.89]],
  "facing": "projection_screen",
  "scale": 0.82,
  "score": 0.91,
  "reasons": ["正对幕布", "不遮挡右侧通道", "与地面透视一致"]
}
```

程序需要执行：

- 排除门口、楼梯口、屏幕前、主要通道和过窄区域。
- 检查与需要保留家具的遮挡关系。
- 按地面四边形和消失方向完成沙发透视缩放与旋转。
- 根据候选区域的远近控制沙发大小，禁止使用固定像素尺寸。
- 输出一张“初版摆位合成图”，用于后续精修和调试。

### 3.5 旧家具移除与局部精修

如果用户要求移除原沙发，先只对旧沙发的遮罩区域做局部修复，再合成新沙发。

精修时使用遮罩限制编辑范围：

- 新沙发边缘：修复锯齿、环境反射、接触关系。
- 地面区域：生成接触阴影和柔和反射。
- 旧沙发区域：补全地毯、墙面、地面或背景。
- 其他区域锁定，避免 AI 改变建筑结构和原有家具。

这一步仍可使用 Gemini 图片编辑，但提示词应明确为“只修复遮罩区域，不改变遮罩外内容”。复杂任务优先使用 `gemini-3-pro-image`；普通精修可使用 Flash Image。

### 3.6 结果质检与重试

每次生成后，用视觉模型对“房间原图、沙发图、结果图、用户要求”进行检查，输出：

```json
{
  "passed": false,
  "issues": [
    "新沙发没有正对投影幕布",
    "右侧通道被扶手遮挡"
  ],
  "correctionPrompt": "将沙发旋转至座面正对投影幕布，并向左移动，保持右侧楼梯通道完整。"
}
```

未通过时，基于结果图进行一次局部纠正；最多重试 1 到 2 次，避免无限消耗积分。

## 4. 分阶段推进建议

### 第一阶段：提升现有生图效果

- 强制结构化解析用户要求。
- 将用户要求、场景分析和每个视角拆成独立生成上下文。
- 增加结果质检和一次纠正重试。
- 在界面展示“模型理解的摆放方案”，允许用户修改后再生成。

优点：改造快，不新增复杂图像基础设施。

### 第二阶段：接入抠图和候选摆位

- 接入沙发分割与透明图输出。
- 输出场景地面区域、障碍物、通道和候选位。
- 提供程序化初版摆位预览。

优点：位置、尺寸和朝向首次变得稳定可控。

### 第三阶段：局部精修与质检闭环

- 使用遮罩进行旧家具移除、阴影和边缘精修。
- 加入结果质检与自动纠正。
- 保存用户偏好和成功摆位，逐渐形成别墅客厅场景规则库。

优点：提高高价值商用素材的稳定性和可复用性。

## 5. 产品交互建议

- 默认由 AI 自动推荐摆位，不让用户面对大量下拉选项。
- 在分析完成后展示一句可编辑的“AI 理解”：
  - 例如：“我将移除原有黑色沙发，保留茶几与地毯，把新沙发摆在投影幕布对面，并留出右侧通道。”
- 用户只需回复“确认”或直接修改文字。
- 高级用户可打开“调整摆位”查看候选位和初版预览。
- 生成失败时明确显示实际模型、服务状态和是否已降级，避免用户误以为图片仍在生成。

## 6. 成功标准

- 用户明确位置要求时，结果满足位置、朝向和保留/移除家具要求。
- 用户未明确位置时，摆位不遮挡通道、屏幕、门窗和主要功能区。
- 新沙发款式、颜色和结构与商品图保持一致。
- 结果图不再原样返回房间图。
- 可通过质检日志追踪“模型为何选择该摆位、为什么重试”。
